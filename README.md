# collect.exe 사용 안내서

## 1. 이 프로그램이 하는 일

GitHub에서 패키지 매니저(npm, yarn 등)별로 오픈소스 프로젝트를 자동으로 찾아서 zip 파일로
내려받고, 그 결과를 표(CSV 파일)로 정리해 주는 프로그램입니다. 어떤 프로젝트를 몇 개 받을지만
정해서 실행하면, 조건에 맞는 프로젝트를 알아서 골라 다운로드합니다.

별도의 설치 없이 `collect.exe` 파일 하나만 있으면 실행됩니다. (Python 등은 필요 없습니다.)

---

## 2. 시작하기 전에 준비할 것

- [ ] **GitHub 계정** (이미 있다고 가정합니다)
- [ ] **GitHub 액세스 토큰(Access Token)** — 아래 방법으로 발급
- [ ] (dotnet 옵션을 쓸 때만) **.NET SDK 설치** — 아래 설명 참고

### 2-1. 액세스 토큰 발급 방법

토큰은 프로그램이 GitHub에 접속할 때 쓰는 "출입증"입니다. 비밀번호처럼 취급해야 합니다.

1. 웹브라우저에서 [https://github.com/settings/tokens](https://github.com/settings/tokens) 로 접속합니다.
2. **Generate new token** → **Generate new token (classic)** 를 클릭합니다.
3. **Note**(메모)에 알아볼 이름을 적고(예: `collect-tool`), **Expiration**(만료일)을 정합니다.
4. 권한(Select scopes)에서 **`public_repo`** 항목에만 체크합니다. (공개 저장소 읽기 전용 권한)
5. 맨 아래 **Generate token** 버튼을 누릅니다.
6. 화면에 `ghp_...`로 시작하는 토큰 문자열이 나옵니다.
   - **이 토큰은 딱 한 번만 보여줍니다.** 안전한 곳(메모장 등)에 복사해 두세요.
   - **남에게 공유하거나 화면 공유 중에 노출하지 마세요.**

### 2-2. dotnet 옵션을 쓸 경우

`--pm dotnet` 옵션은 실제로 프로젝트를 빌드해 보는 과정이 있어서, 실행하는 컴퓨터에
**.NET SDK가 설치되어 있어야만** 동작합니다.
([https://dotnet.microsoft.com/download](https://dotnet.microsoft.com/download) 에서 설치)

- 나머지 8개(npm, yarn, maven, gradle, go, composer, dart, bundler)는 .NET SDK가 필요 없습니다.
- .NET SDK가 없으면 dotnet 옵션만 실패하고, 다른 옵션에는 영향이 없습니다.

## 3. 실행 방법

### 3-1. 토큰을 환경변수로 등록

PowerShell 창을 열고, 발급받은 토큰으로 아래 명령을 실행합니다.
(`ghp_...` 부분을 본인 토큰으로 바꿔 넣으세요.)

```powershell
$env:GITHUB_TOKEN = "ghp_여기에_본인_토큰_붙여넣기"
```

> 참고: 이 설정은 **지금 열어둔 PowerShell 창에서만** 유효합니다. 창을 닫으면 사라지므로,
> 새 창을 열면 다시 입력해야 합니다.

### 3-2. 프로그램 실행

`collect.exe`가 있는 폴더에서, 같은 PowerShell 창에 아래처럼 입력합니다.

```powershell
.\collect.exe --pm npm --count 5
```

위 예시는 "npm 프로젝트 5개를 받아라"라는 뜻입니다.

### 3-3. "알 수 없는 게시자" 경고가 뜰 때

처음 실행하면 Windows가 파란 화면으로 **"Windows의 PC 보호"** 경고를 띄울 수 있습니다.
직접 만든 프로그램이라 서명이 없어서 그런 것이니, 아래처럼 진행하면 됩니다.

1. **추가 정보** 클릭
2. **실행** 버튼 클릭

## 4. 명령줄 옵션 설명

| 옵션 | 하는 일 | 기본값 | 예시 |
|------|---------|--------|------|
| `--pm` | 어떤 패키지 매니저를 수집할지 (필수) | (없음, 반드시 지정) | `--pm npm` |
| `--count` | 매니저별로 받을 프로젝트 개수 | `5` | `--count 30` |
| `--min-stars` | 이 별점(star) 미만인 프로젝트는 제외 | `0` | `--min-stars 50` |
| `--max-size-kb` | 이 크기(KB)보다 큰 프로젝트는 제외 (너무 큰 저장소 거르기) | `300000` (약 300MB) | `--max-size-kb 100000` |
| `--out` | 결과를 저장할 폴더 | `C:\Exception\ScraperTest` | `--out D:\내수집폴더` |
| `--verbose` | 실행 과정을 더 자세히 화면에 표시 | (꺼짐) | `--verbose` |
| `--candidate-multiplier` | 목표 개수를 채우기 위해 몇 배수의 후보를 검색할지 (보통 그대로 둡니다) | `4` | `--candidate-multiplier 6` |

### `--pm`에 넣을 수 있는 값

```
npm, yarn, maven, gradle, go, composer, dart, bundler, dotnet, all
```

`all`을 넣으면 위 9개 매니저를 모두 수집합니다.

---

## 5. 사용 예시

먼저 토큰을 등록한 뒤(3-1 참고), 아래 명령들을 그대로 복사해서 쓰면 됩니다.

**npm 프로젝트를 30개 받고 싶을 때**

```powershell
.\collect.exe --pm npm --count 30
```

**특정 폴더에 저장하고 싶을 때**

```powershell
.\collect.exe --pm npm --count 30 --out D:\수집결과
```

**모든 매니저를 한 번에 수집하고 싶을 때**

```powershell
.\collect.exe --pm all --count 10
```

**실행이 잘 안 돼서 무슨 일이 벌어지는지 자세히 보고 싶을 때**

```powershell
.\collect.exe --pm npm --count 5 --verbose
```

## 6. 실행하면 무슨 일이 일어나나요

프로그램은 대략 아래 순서로 동작합니다. 그래서 프로젝트 개수가 많으면 시간이 걸립니다.

- GitHub에서 조건에 맞는 후보 프로젝트를 검색합니다.
- 각 후보가 조건에 맞는지 확인합니다. (저장소 크기, 진짜 프로젝트 설정 파일이 있는지, 라이브러리
  캐시 폴더만 잔뜩 든 건 아닌지 등)
- 조건을 통과한 프로젝트만 zip 파일로 내려받습니다. (조건에 안 맞으면 건너뜁니다 — 그래서 검색은
  넉넉히 하고 그중 일부만 최종 수집됩니다.)
- **dotnet의 경우에만**, 내려받은 뒤 실제로 빌드를 한 번 돌려서 정상 프로젝트인지 검증하는
  추가 단계가 있습니다. (이 과정 때문에 dotnet은 다른 것보다 오래 걸립니다.)
- 최종적으로 각 프로젝트의 정보를 CSV 로그 파일에 정리해서 기록합니다.

검색은 GitHub의 속도 제한이 있어서, 중간에 잠깐 기다리는 구간이 생길 수 있습니다. (정상입니다.)

## 7. 결과물은 어디서 확인하나요

`--out`으로 지정한 폴더(기본값 `C:\Exception\Scraper`) 아래에 결과가 저장됩니다.

- **다운로드된 zip 파일**: `data\{매니저이름}\` 폴더 안에 저장됩니다.
  예) `C:\Exception\ScraperTest\data\npm\npm_facebook_react.zip`
- **정리된 결과표**: `collection_log.csv` 파일에 프로젝트마다 한 줄씩 기록됩니다.
  - 별점, 크기, 성공/실패 여부, (실패했다면) 실패 사유 등이 들어 있습니다.
  - **엑셀로 열어서** 확인할 수 있습니다.

## 8. 문제가 생겼을 때

**"401" 또는 "Bad credentials" 오류가 뜨면?**
- 토큰이 잘못 입력됐거나, 환경변수가 등록되지 않은 경우입니다.
- 3-1 단계를 다시 실행해 토큰을 등록했는지, 토큰 문자열이 정확한지(앞뒤 공백/따옴표 확인) 다시
  확인하세요. 토큰 만료일이 지났을 수도 있으니, 필요하면 새로 발급하세요.

**실행이 너무 오래 걸리면?**
- `dotnet`은 실제 빌드 과정이 있어서 원래 가장 오래 걸립니다. 멈춘 게 아니라 정상입니다.
- 진행 상황을 보고 싶으면 `--verbose` 옵션을 붙여 실행하세요.
